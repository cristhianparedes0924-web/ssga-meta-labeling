Section 03.5 — Benchmark Comparison Suite

Who Is M1 Competing Against?

A timing model is only as good as the alternatives it beats. Four benchmarks, each answering a different question about M1's value.

BENCHMARK 0

Buy & Hold SPX

Always long. Earns every up month and every down month.

Question: Does timing add value over passive equity exposure?
BENCHMARK 1

60/40 Portfolio

60% S&P 500, 40% 10Y Treasury. Passive diversification.

Question: Does timing beat simple diversification?
BENCHMARK 2

10M vs 12M moving average on SPX price. Single-indicator trend.

Question: Do 5 indicators beat 1 simple price rule?
BENCHMARK 3

Random Signals

1,000 Monte Carlo simulations with same BUY/SELL/HOLD distribution.

Question: Is M1's edge statistically real or luck?

Why These Three (Beyond Buy & Hold)?

Together, these benchmarks form a hierarchy of challenge. The 60/40 tests if M1 beats the simplest institutional allocation. The SMA tests if M1's multi-indicator complexity is justified — an ablation test isolating whether the 4 non-momentum indicators (cross-asset, yield, credit, vol) add value beyond trend-following alone. The random baseline establishes statistical significance: even if M1 beats the first two, we need to know the probability that a monkey throwing darts could match M1's Sharpe ratio. Only if M1 passes all three can we confidently say its signals contain genuine predictive information worth feeding to M2.

map_signals_to_positions(df)

Converts text signals to numerical position values
▼
💡 Why This Was Needed

Your original code generated text signals (BUY/SELL/HOLD) but never converted them into numbers. A backtest requires numerical positions (+1, 0, -1) so you can multiply by returns to simulate P&L.

✅ Design Decisions
  • Long-only (SELL → 0): Your indicators signal when to be in or out of equities, not when to short. A long_only flag is included for later toggling.
  • HOLD → forward-fill: "Don't change anything" = carry previous position via ffill().
  • Initial = 0: Flat before first signal fires (warmup period).
📄 Paper Reference

Joubert's primary model (Exhibit 1) outputs ŷ ∈ {-1, 0, 1}. The long-only constraint {0, 1} matches the paper's note that models can be "long only {0,1}" (p. 32).

Python
def map_signals_to_positions(df):
    long_only = True
    df = df.copy()
    signal_map = {
        'BUY':  1,
        'SELL': 0 if long_only else -1,
        'HOLD': np.nan   # will forward-fill
    }
    df['Position'] = df['Signal'].map(signal_map)
    df['Position'] = df['Position'].ffill()
    df['Position'] = df['Position'].fillna(0)
    return df

calculate_strategy_returns(df)

Computes strategy P&L with look-ahead bias prevention
▼
💡 Why This Was Needed

This is the core of the backtest. The critical nuance is timing: you observe a signal at end of month t, take the position, earn the return in month t+1. Getting this wrong is the most common backtest error.

✅ Design Decisions
  • shift(-1): Standard look-ahead bias prevention. Signal at t uses data at t; return earned at t+1.
  • Strategy Return = Position × Forward Return: Long (+1) earns full return; flat (0) earns nothing.
  • Buy-and-hold baseline: Always-long returns as benchmark. If strategy can't beat this risk-adjusted, it adds no value.
  • (1 + return).cumprod(): Growth of $1. Compounds correctly unlike cumulative sum.
Python
def calculate_strategy_returns(df):
    df = df.copy()
    df['Forward_Return'] = df['SPX_Return'].shift(-1)  # ← prevents look-ahead
    df['Strategy_Return'] = df['Position'] * df['Forward_Return']
    df['BH_Return'] = df['Forward_Return']
    df['Strategy_Cumulative'] = (1 + df['Strategy_Return'].fillna(0)).cumprod()
    df['BH_Cumulative']       = (1 + df['BH_Return'].fillna(0)).cumprod()
    return df

calculate_performance_metrics(df)

Sharpe ratio, max drawdown, Calmar, win rate, trade count
▼
💡 Why This Was Needed

Your original code had no way to evaluate whether signals were any good. This computes the standard suite of metrics any quant team would expect. The Sharpe ratio is the key metric meta-labeling aims to improve.

✅ Design Decisions
  • Geometric annualization: (cumulative)^(1/years) - 1 is more accurate than simple avg × 12 because it accounts for compounding and volatility drag.
  • Sharpe with rf = 0: Simplicity. For comparing Strategy vs B&H, risk-free cancels out anyway.
  • Max drawdown from expanding max: Tracks running peak, measures worst decline. This metric makes or breaks real strategies.
  • Trade counting via Position.diff(): Counts actual position changes. Per the paper's Eq 1-2, more trades = more sensitivity to precision = more room for M2.
  • Time in market: If you're invested only 30% of time but matching B&H, risk-adjusted performance is excellent.
📄 Paper Reference

Joubert's Equation 1: θ(p, n, π₋, π₊) — Sharpe as function of precision, trades, payouts. "The first step [before meta-labeling] is to understand the strategy's risks" (p. 34). This function computes all those parameters.

Python — Key Logic
# Annualized return (geometric)
cumulative = (1 + returns).prod()
ann_return = cumulative ** (1 / n_years) - 1

# Max drawdown
cum_series = (1 + returns).cumprod()
rolling_max = cum_series.expanding().max()
max_dd = ((cum_series - rolling_max) / rolling_max).min()

# Trade count
position_changes = (df['Position'].diff() != 0).sum()

calculate_signal_accuracy(df)

Precision, recall, F1 — classification metrics for signals
▼
💡 Why This Was Needed

The entire meta-labeling framework revolves around the precision-recall trade-off. You need your current precision and recall before M2, because M2's job is specifically to trade recall for precision. Without a baseline, you can't measure improvement.

✅ Design Decisions
  • Precision = correct BUYs / total BUYs: Drives Sharpe per the paper. "True positives are rewarded, false positives are punished, negatives are not rewarded" (p. 34).
  • Recall = caught up-months / all up-months: Paper says build M1 with high recall so M2 can correct low precision.
  • F1 = harmonic mean: Single number for precision-recall balance. M2 should improve this.
  • HOLD excluded: No position taken = no prediction to judge.
📄 Paper Reference

"Fitting a primary model that has a high recall is an important step… so that the low precision can be corrected by the secondary model, resulting in a higher F1-score" (p. 33). "The Sharpe ratio is a function of precision rather than accuracy" (p. 34).

Python — Key Logic
# Precision
buy_signals = valid[valid['Signal'] == 'BUY']
buy_precision = (buy_signals['Forward_Return'] > 0).sum() / len(buy_signals)

# Recall
positive_periods = valid[valid['Forward_Return'] > 0]
recall = len(positive_periods[positive_periods['Signal'] == 'BUY']) / len(positive_periods)

# F1
f1 = 2 * (buy_precision * recall) / (buy_precision + recall)

generate_meta_labels(df)

Creates the target variable for M2 — Equation 8 from the paper
▼
💡 Why This Was Needed

Meta-labels are the bridge between M1 and M2. They answer: "Was M1's prediction correct?" When you build M2, it trains on these to learn when M1 is right vs wrong. Generating them now saves rebuilding the backtest later.

✅ Design Decisions
  • Binary {0, 1}: Paper defines meta-labels as binary — profitable (1) or not (0). No partial credit.
  • Direction-based: BUY correct if return > 0, regardless of magnitude. Matches Equation 8.
  • HOLD → NaN: No bet = no label. M2 trains only on active signals.
  • Generated before split: Labels use forward returns already computed. Split happens after.
📄 Paper Reference

Equation 8 (p. 37): "y_t = 1 if ẏ_t = sign(Δr_{t+1}) = 1, else 0" — meta-label is 1 when primary signal matches actual direction.

Python
def generate_meta_labels(df):
    df = df.copy()
    df['Meta_Label'] = np.nan
    # BUY correct if next return positive
    df.loc[(df['Signal']=='BUY') & (df['Forward_Return']>0),  'Meta_Label'] = 1
    df.loc[(df['Signal']=='BUY') & (df['Forward_Return']<=0), 'Meta_Label'] = 0
    # SELL correct if next return negative
    df.loc[(df['Signal']=='SELL') & (df['Forward_Return']<0),  'Meta_Label'] = 1
    df.loc[(df['Signal']=='SELL') & (df['Forward_Return']>=0), 'Meta_Label'] = 0
    return df  # HOLD rows stay NaN

calculate_rolling_metrics(df, window=12)

Rolling Sharpe and hit rate — future features for M2
▼
💡 Why This Was Needed

The paper's Exhibit 2 shows M2 needs "evaluation data" (e₁, e₂, ..., eₖ) measuring M1's recent health. When rolling Sharpe drops or hit rate falls below 50%, M1 is entering a bad regime. These are computed now so they're ready for M2.

✅ Design Decisions
  • 12-month window: Matches the longest lookback in your indicators (12M momentum). Stable enough yet responsive.
  • Rolling Sharpe annualized: Same metric as strategy-level, just trailing. mean × 12 / (std × √12).
  • Rolling hit rate: Fraction of recent signals correct. Sustained drop below 50% = M1's edge has faded.
📄 Paper Reference

"Evaluation statistics based on the primary model should be included to help indicate when recent performance is poor. Relevant concepts include rolling accuracy, F1, recall, precision, and AUC scores." (p. 40)

Python
def calculate_rolling_metrics(df, window=12):
    df = df.copy()
    rolling_mean = df['Strategy_Return'].rolling(window).mean() * 12
    rolling_std  = df['Strategy_Return'].rolling(window).std() * np.sqrt(12)
    df['Rolling_Sharpe'] = rolling_mean / rolling_std

    # Hit rate: % of recent signals correct
    df['Signal_Hit'] = np.nan
    # ... (label each signal as correct/incorrect)
    df['Rolling_HitRate'] = df['Signal_Hit'].rolling(window, min_periods=1).mean()
    return df

train_test_split(df, train_ratio=0.60)

60/40 sequential time-series split — no shuffling
▼
💡 Why This Was Needed

Even though M1 is rule-based, you need to check generalization. If you tuned thresholds by looking at the full sample, you're implicitly overfitting. And when M2 comes, this split is essential — M2 literally trains on the train set.

✅ Design Decisions
  • 60/40 ratio: Directly from the paper: "split into a train/test set of 60/40 without shuffling" (p. 36).
  • Sequential (no shuffle): Time series has temporal dependencies. Shuffling = future data leak.
  • Split on valid-signal rows: Excludes warmup period. Pre-signal rows tagged as 'pre-signal'.
  • 'Set' column persists: Every row tagged for filtering in CSV and future M2 work.
📄 Paper Reference

"The data are then split into a train/test set of 60/40 without shuffling, that is, the most recent 40% of the data are held in the test set." (p. 36)

Python
def train_test_split(df, train_ratio=0.60):
    valid_df = df.dropna(subset=['Signal']).copy()
    split_idx = int(len(valid_df) * train_ratio)

    df['Set'] = 'pre-signal'
    df.loc[... first 60% dates ...,  'Set'] = 'train'
    df.loc[... last 40% dates ...,   'Set'] = 'test'

    train_df = df[df['Set'] == 'train'].copy()
    test_df  = df[df['Set'] == 'test'].copy()
    return df, train_df, test_df

calculate_benchmark_6040(df)

Benchmark 1 — 60% S&P 500 / 40% 10Y Treasury passive portfolio
▼
💡 Why This Benchmark

The 60/40 portfolio is the institutional default. Every timing model must answer one question first: does active management beat passive diversification? Our M1 model switches between 100% equity and 0% equity based on macro signals. If that timing cannot beat a static 60/40 allocation, the signals are not adding value beyond the simplest diversification approach — and there is no justification for the additional complexity.

✅ Design Decisions
  • 60% SPX + 40% Treasury: Uses the same data already loaded (S&P 500 returns + 10Y Treasury returns). No additional data dependencies.
  • Static weights: The portfolio uses fixed 60/40 weights each month. Monthly rebalancing effects are negligible at this frequency, and static weights avoid introducing any implicit timing.
  • Same forward-return convention: Uses shift(-1) on both SPX and Treasury returns to maintain identical timing with the strategy — no look-ahead advantage or disadvantage.
  • Blended return formula: r = 0.60 × r_SPX + 0.40 × r_Treasury. Simple weighted average, computed each period.
📄 Context

While Joubert (2022) uses Buy-and-Hold as the primary benchmark, institutional practice requires comparison against a balanced portfolio. The 60/40 is particularly relevant because it captures the diversification benefit M1 is competing against: Treasury bonds tend to rise when equities fall, providing a natural hedge that a binary equity timing model does not.

Python
def calculate_benchmark_6040(df):
    df = df.copy()
    # Forward returns for both assets (same timing convention)
    fwd_spx = df['SPX_Return'].shift(-1)
    fwd_treasury = df['Treasury10Y_Return'].shift(-1)

    # 60/40 blended return
    df['Bench_6040_Return'] = 0.60 * fwd_spx + 0.40 * fwd_treasury

    # Cumulative growth of $1
    df['Bench_6040_Cumulative'] = (1 + df['Bench_6040_Return'].fillna(0)).cumprod()
    return df

calculate_benchmark_sma(df)

Benchmark 2 — 10M/12M Simple Moving Average crossover on SPX
▼
💡 Why This Benchmark

M1's strongest indicator is 12-month price momentum (Indicator 1). This benchmark isolates a simple trend-following rule using only price. If M1 cannot outperform a basic SMA crossover, then the other 4 indicators (cross-asset, yield change, credit, volatility regime) are adding noise rather than signal. This is an ablation test: does multi-indicator complexity beat single-indicator simplicity?

✅ Design Decisions
  • 10M vs 12M SMA pair: A standard medium-term trend pair from Faber (2007). The 10-month SMA is the classic "Ivy Portfolio" signal; pairing with 12M creates a crossover that filters noise better than a single threshold.
  • Matches M1's lookback range: Our indicators use 6M–12M windows. Using 10/12 makes the comparison fair — same information horizon.
  • Long when SMA_10 > SMA_12, flat otherwise: Same long-only constraint as M1. When short-term trend is above long-term trend, market has positive momentum → stay long. When it crosses below → step aside.
  • Same position → forward-return logic: Position determined at time t, return earned at t+1. Identical to M1's convention.
📄 Context

Faber (2007), "A Quantitative Approach to Tactical Asset Allocation," demonstrated that a 10-month SMA rule on the S&P 500 achieved equity-like returns with bond-like volatility. This has become a standard trend benchmark in tactical allocation research. If M1's multi-factor approach doesn't beat this simple rule, the additional factors need revision.

Python
def calculate_benchmark_sma(df, short_window=10, long_window=12):
    df = df.copy()
    # Compute SMAs on SPX price
    df['SMA_Short'] = df['SPX_Price'].rolling(short_window).mean()
    df['SMA_Long']  = df['SPX_Price'].rolling(long_window).mean()

    # Long when short SMA > long SMA (uptrend), flat otherwise
    df['SMA_Position'] = np.where(df['SMA_Short'] > df['SMA_Long'], 1, 0)

    # Apply to forward returns
    fwd_spx = df['SPX_Return'].shift(-1)
    df['Bench_SMA_Return'] = df['SMA_Position'] * fwd_spx
    df['Bench_SMA_Cumulative'] = (1 + df['Bench_SMA_Return'].fillna(0)).cumprod()
    return df

calculate_benchmark_random(df)

Benchmark 3 — Monte Carlo random signal baseline (1,000 simulations)
▼
💡 Why This Benchmark

This is a statistical validity test. If M1's performance cannot be distinguished from randomly generated signals with the same BUY/SELL/HOLD distribution, then M1 has no real predictive edge — any apparent outperformance is attributable to luck. This is particularly critical for the meta-labeling framework: M2's job is to filter M1's signals, but if M1 is not better than random to begin with, M2 has nothing meaningful to learn from.

✅ Design Decisions
  • Same signal distribution: Random signals match M1's empirical frequency (e.g., if M1 generates 45% BUY / 20% SELL / 35% HOLD, random does the same). This ensures any performance difference comes from timing, not from being long more often.
  • 1,000 simulations: Enough for stable percentile estimates. The 95th percentile of 1,000 Sharpes gives a reliable p < 0.05 threshold.
  • Same position logic: Random HOLD → forward-fill, same as M1. Ensures apples-to-apples comparison of the position mapping.
  • Median simulation as benchmark line: The 500th-ranked simulation (by Sharpe) becomes the plotted benchmark. This represents "what average luck looks like."
  • Full Sharpe distribution stored: df.attrs['random_sim_sharpes'] stores all 1,000 Sharpes for the significance test function.
  • seed=42: Reproducibility. Same random sequences every run.
📄 Context

Monte Carlo permutation testing is standard in quantitative finance for establishing statistical significance. White (2000), "A Reality Check for Data Snooping," and later Hansen (2005) formalize this approach. In the meta-labeling context, if M1 passes this test (p < 0.05), it confirms M1 has genuine signal that M2 can learn to amplify. If it fails, the priority should be improving M1's indicators before building M2.

Python — Key Logic
def calculate_benchmark_random(df, n_simulations=1000, seed=42):
    # Get M1's signal distribution
    signal_counts = valid_signals['Signal'].value_counts(normalize=True)

    for i in range(n_simulations):
        # Generate random signals with same BUY/SELL/HOLD frequencies
        random_signals = rng.choice(signal_labels, size=n_periods, p=signal_probs)

        # Map to positions (same HOLD=ffill logic)
        for t in range(n_periods):
            if random_signals[t] == 'HOLD':
                positions[t] = positions[t-1]  # carry forward

        # Compute Sharpe for this simulation
        sharpe = (sim_returns.mean() * 12) / (sim_returns.std() * np.sqrt(12))
        sim_sharpes.append(sharpe)

    # Use median simulation as the benchmark line
    median_idx = np.argsort(sim_sharpes)[n_simulations // 2]

print_random_significance(df, strategy_sharpe)

Statistical significance test — M1 Sharpe vs random Sharpe distribution
▼
💡 Why This Was Needed

Having 1,000 random Sharpes is useless without a formal comparison. This function computes M1's percentile rank in the random distribution and reports a one-tailed p-value. It answers: "What is the probability that M1's Sharpe could have been achieved by random timing?"

✅ Design Decisions
  • Percentile rank: Fraction of random Sharpes below M1's Sharpe. If M1 is at the 97th percentile → only 3% of random strategies did as well.
  • One-tailed p-value: p = 1 - percentile/100. We only care if M1 is significantly better than random, not different in either direction.
  • Three-tier reporting: p < 0.05 = significant ✅, p < 0.10 = marginal ⚠️, p ≥ 0.10 = not significant ❌.
  • Called on test-set Sharpe: The significance test uses out-of-sample performance — the only honest evaluation.
Python
def print_random_significance(df, strategy_sharpe_str):
    sim_sharpes = df.attrs['random_sim_sharpes']
    strategy_sharpe = float(strategy_sharpe_str.split()[0])

    # Percentile rank of M1's Sharpe in random distribution
    percentile = (sim_sharpes < strategy_sharpe).sum() / len(sim_sharpes) * 100
    p_value = 1 - percentile / 100

    if p_value < 0.05:
        print("✅ SIGNIFICANT — M1 outperforms random")
    elif p_value < 0.10:
        print("⚠️  MARGINALLY SIGNIFICANT")
    else:
        print("❌ NOT SIGNIFICANT — could be luck")

Section 05 — What Gets Measured

Metrics Explained

Two categories. Click tabs to switch.

📈

Annualized Return

Geometric annual rate. Accounts for compounding.

(1 + cumulative)^(1/years) − 1
📊

Annualized Volatility

Std dev × √12. Higher = more uncertainty.

σ_monthly × √12
⭐

Sharpe Ratio

Return per unit of risk. Meta-labeling's primary target.

Ann. Return / Ann. Volatility
📉

Maximum Drawdown

Worst peak-to-trough. Meta-labeling aims to reduce.

(trough − peak) / peak
⚖️

Calmar Ratio

Return / |max drawdown|. Higher = better.

Ann. Return / |Max DD|
🎯

Win Rate

% positive periods. Size of wins matters more.

# positive / # total
🎯

Precision (BUY)

Correct BUYs / all BUYs. Drives Sharpe ratio.

TP / (TP + FP)
🔍

Recall (BUY)

Caught up-months / all up-months. M2 trades this for precision.

TP / (TP + FN)
⚡

F1 Score

Harmonic mean. M2 should improve this.

2 × (P × R) / (P + R)
📋

Overall Accuracy

% directionally correct. Less important than precision.

Correct / Total active

Why Precision > Accuracy

True positives earn money, false positives lose money, true negatives don't matter (no position). So precision = correct bets / total bets is what drives risk-adjusted return.

Section 06 — Reading Your Results

What to Look For

✅ Good Signs

• Sharpe > 0 on test set

• Train ≈ test metrics (generalizes)

• BUY precision > 50%

• High recall (room for M2)

• Beats 60/40 and SMA on Sharpe

• Random p-value < 0.05

⚠️ Warning Signs

• Big train/test gap (overfitting)

• Very few trades (M2 needs data)

• High win rate + low Sharpe (big losses)

• Beats B&H but not SMA (extra indicators = noise)

• Random p-value between 0.05–0.10

🚨 Red Flags

• Negative test Sharpe (destroying value)

• Precision < 50% (signals mostly wrong)

• Drawdown worse than B&H

• Can't beat random baseline (p > 0.10)

• Loses to 60/40 on risk-adjusted basis

The Precision–Sharpe Relationship (Exhibit 5)

More trades = steeper slope → small precision improvement → bigger Sharpe jump. This is why the backtest counts position changes. Many trades + mediocre precision = M2 can help a lot.
